## User Clarification

The device is not meant to be a supercomputer.

Its purpose is to act as a stable hub that connects work across different
locations while housing the repository and supporting the same level of work the
user is doing now.

## Updated Decision Frame

The corrected decision is:

```text
What is the most reliable, cost-disciplined work hub for repository-centered
development, documentation, research, and continuity across locations?
```

## Hub Requirements

A strong candidate should support:

- repository hosting and local working copy stability
- Git/GitHub workflows
- Codex and VS Code work
- Microsoft Office and documentation work
- browser-heavy research
- remote or multi-location access patterns
- stable networking
- predictable sleep / wake / uptime behavior
- low-friction backup and sync strategy
- enough storage for active work without forcing constant cleanup
- simple recovery if the device fails

## Reweighted Priorities

Increase emphasis on:

- storage reliability
- network stability
- backup and sync fit
- low idle power and stable always-available behavior
- quiet operation
- warranty / return path
- seller trust
- practical adequacy
- completed cost

Decrease emphasis on:

- maximum CPU performance
- AI branding
- large RAM counts beyond practical need
- premium processors when a midrange CPU is already sufficient
- exceptional-band spend

## Specification Implication

For the updated work-hub framing:

- 32 GB RAM remains the right practical target.
- 1 TB storage is acceptable, but 2 TB becomes more valuable if the price gap is
  reasonable because the machine may hold the repository hub and supporting
  work files.
- 64 GB RAM is less important than storage, backup discipline, and network
  reliability.
- USB4, dual Ethernet, and stable Wi-Fi are valuable because they support hub
  and docking behavior.
- A cheaper machine with weaker warranty/support can still win only if return
  and backup/recovery plans are clear.
